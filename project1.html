2018-09-12 - Second Session

Today I spent more time off-book searching and reading about questions that came up that weren't explained in the book's material. The book is good, but very focused on exam material so I had to look outside for some other sources to get a little more context about concepts that weren't explained in the book.

The remainder of the book material dealt with two concepts. The first was booleans which are predefined sets of policy features that can be quickly turned on and off. These policy groups can be toggled temporarily or more permanantly with some simple commands. There are a lot of these, but one of the status commands at least gives a nice description as to what they do. Searching through those descriptions would let a person narrow down at least to a few sets of policies dealing with the area they are interested in. I suspect this is more commonly used than the individual tags. I need to look and see if there is a way to map these boolean policy groups to individual tags.

The second concept was about diagnosing and addressing policy violations. It described how to find SELinux entries in the log, as well as a troubleshooting package that writes additional information to /var/log/messages. Somehow they don't print out as nicely in my terminal as they do in the book. At the start of the text blob is a message saying to run sealert -l [some uuid] to get complete information. Running this gives a much nicer-formatted output along with additional info to what is in the book listing.

Outside the book one of the questions that I looked into was the s0 entries at the end of each label. It turns out that they are called sensitivity and there is another type as well called category levels (they look like s0, s3, c0, c3, etc.) Still after reading that I was unclear on what they do, as the document mentioned they were involved with some things called MLS and MCS. After digging some more I see that they are used for two different enhancement packages for SELinux called MCS and MLS, which stand for Multi Category Security and Multi Level Security. So this explains why all these basic exercises show s0 for everything since the material that I'm looking at doesn't involve these packages.

  • https://access.redhat.com/documentation/en-us/red_hat_enterprise_linux/6/html/security-enhanced_linux/chap-security-enhanced_linux-selinux_contexts
  • https://access.redhat.com/documentation/en-us/red_hat_enterprise_linux/5/html/deployment_guide/sec-mcs-ov
  • https://access.redhat.com/documentation/en-us/red_hat_enterprise_linux/7/html/selinux_users_and_administrators_guide/mls

Another item that I looked into was the question of how these labels interact with remote shares like NFS. How, in that situation do two different systems manage their labels?

  • https://access.redhat.com/documentation/en-us/red_hat_enterprise_linux/7/html/selinux_users_and_administrators_guide/sect-security-enhanced_linux-working_with_selinux-mounting_file_systems

They mention:
In this example, server:/export/web is mounted locally to the /local/web/ directory, with all files being labeled with the httpd_sys_content_t type, allowing Apache HTTP Server access. server:/export/database is mounted locally to /local/database/, with all files being labeled with the mysqld_db_t type, allowing MariaDB access. These type changes are not written to disk.

So sounds like the labeling is perhaps not written back to the file on an nfs share? Need to find some more sources of informaiton on this.

Finally, I came across this talk on SELinux from Red Hat's Thomas Cameron. I've been to some of his talks before and he's a great presenter, so need to watch this one and see if he has any additional insights or comes at it from another angle https://www.youtube.com/watch?v=cNoVgDqqJmM

Generally I've been noticing that when I have a question about how some aspect of it works or relates, the Red Hat documentation has been where I ultimately find the answer. Not always in the place or under the heading I expect or first seem to look, but the documentation does seem to be very good.

2018-09-05 - First Session Progress

Today I read through the first few sections of the van Vugt book. Covered some definitions, how to enable/disable the whole SELinux system, both temporarily and from startup. Next was a more in-depth look at context settings including how to view them, the notation of what they mean, how to change the type tags in the policy and applying that policy to the files which takes the tags defined in the policy and sets them in the files.

The content so far focused more on setting the type tags than listing the different type tags and what they mean, but did include some info about looking up the different ones in man. I may need to go looking for some list of the most common ones or something like that to get familiar with at least some key subset and get an idea about how they are classified.

Another thing I need to look up is what the :s0 notation is at the end of the context label. It is there in all of the examples but not really explained as he is staying focused on the type tags, so I need to look that up.

One unexpected offshoot was in one of the exercises he has us use the elinks browser which is apparently an offshoot of lynx. Years ago I wanted to get comfortable with lynx but the version packaged with my system had problems running. Elinks seems to run fine though, so I can keep that around to play with later.

At the end of the section a question that bothered me was where are all these labels stored, which could be important when it comes to things like backing up. That led me to find this discussion where someone was wondering the same thing regarding SELinux and rsync: https://forums.fedoraforum.org/showthread.php?276028-SELinux-how-where-labels-stored-securing-rsync-backup

The takeaway for my question was that the labels are stored with the file as an attribute. The discussion also had a good point about what those labels mean when it comes to the backup/storage system which may require a different policy. That leads to the question of policy storage which I found some answers to here: https://www.centos.org/docs/5/html/Deployment_Guide-en-US/rhlcommon-chapter-0001.html That page described the locations where the policy information is stored. Given the Van Vugt book's emphasis on the policy being the master I think it would make the most sense to back up both the files and the policy, and not treat the labels on the files as definitive, but instead reapply the policy after a restore. That would also allow a different policy to be applied on the backup storage server.
